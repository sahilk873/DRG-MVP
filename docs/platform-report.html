1What it is

A trustworthy layer between the medical record and the claim.

Hospitals lose revenue and take on compliance risk when the clinical record, the coded claim, and the billed charges disagree. Encounter reconstructs each inpatient stay from a provider's own data, reconciles it against the claim, and surfaces only the discrepancies a qualified person actually needs to decide — a missed diagnosis, an unsupported charge, a present-on-admission conflict.

The defining design choice is determinism: language models help interpret variable, messy inputs, but nothing a model produces can execute, change a claim, assign a DRG, compute a payment, or bypass review. Every consequential result is reproducible, versioned, and hash-audited. That is what makes the output defensible to a coder, a compliance officer, and an auditor.

Three cooperating parts

Python engine — deterministic models, rules, grouper, audit.
Mastra/TS agents — semantic extraction & authoring only.
React demo — the interactive pitch & reviewer UI.

Who it serves

Hospital revenue-integrity, CDI (clinical documentation integrity), coding, and compliance teams — plus the finance leaders who care about recoverable, defensible revenue.

What makes it different

Bi-directional (catches over- and under-coding), evidence-grounded to an exact source excerpt, and reproducible end to end — not a black box.

2The core invariant

The single rule the whole architecture protects.

Core invariant No language-model output may execute code, create or change a claim, assign a DRG, compute reimbursement, or bypass required review. The agent emits only evidence excerpts and schema-constrained hypotheses; deterministic Python then verifies exact grounding, merges immutable source fields, evaluates governed declarative rules, and calls a licensed-grouper boundary.

This is enforced structurally, not by convention, and it is locked by an executable test suite:

  • Rules are a declarative JSON DSL (a fixed operator + change-key set) — there is no eval/exec of generated code on the claim path.
  • The grouper/pricer is a Protocol boundary; the shipped grouper is a fake whose version literally contains not-for-billing.
  • Rule and adapter packages fail closed unless status ∈ {approved, approved-for-demo}.
  • The review packet sets claim_mutation_allowed = false; any claim-affecting finding forces requires_human_review.
  • Money is integer-cent only; IDs are deterministic; every result carries full version provenance and a hash chain.

There are two trust boundaries, and each keeps model influence to interpretation only:

① Adapter factory (onboarding)

The adapter-designer agent sees only a bounded profile of a bulk export and proposes a declarative mapping. The deterministic runtime sees the full data and accepts only an approved, fingerprint-compatible adapter. Discovery and execution never share a code path.

② Evidence extraction

The extractor agent receives encounter timing, an ontology contract, and source documents — but not the claim. It returns evidence excerpts + an ontology fragment + assertions; Python verifies every excerpt is a literal source substring, checks the ontology, merges the immutable claim, and runs an independent validator.

3End-to-end pipeline

From a provider's raw export to one governed human decision.

Each stage below expands one box. The colored responsibility split is the whole story: agents interpret variable inputs, deterministic code produces every consequential result, and a person owns the final call.

4Bulk onboarding — the adapter factory

src/revenue_integrity/ingestion/

Providers export data in wildly different shapes. Onboarding is a governed three-step boundary:

  1. Profile (profiling.py) — a deterministic profiler reads the export folder and emits a bounded profile: column names, inferred types, tiny samples, a schema_fingerprint (columns/types) and an input_manifest_digest (file content). The full dataset never leaves the deterministic plane.
  2. Design (agents/adapter-designer.ts) — a Mastra agent sees only that profile and proposes a declarative adapter (a JSON mapping): resource files, field bindings, transform ops, row filters, and structured ontology projections. It is explicitly forbidden to emit Python/JS/SQL/regex/executable code.
  3. Run (adapter.py, readers.py, transforms.py) — a human approves the adapter; the deterministic runtime validates the fingerprint matches (schema-drift check), then transforms the full dataset into canonical source bundles with row-level provenance (adapter id/version, resource, path, row number, source-record id, field names).

Transform operations (the fixed, safe DSL)

Readers cover CSV · JSON · JSONL · XLSX. Field transforms are a deliberately narrow set — trim, lower, upper, integer, number, boolean, datetime (with format + timezone), map (fail-closed enum), split. Structured projections deterministically build ontology entities/relations/assertions from source rows (no model inference), each grounded to an evidence locator.

Graceful degradation (quarantine plane)

By default any bad row fails the whole batch closed. Opt in with a DegradationPolicy(mode="quarantine") and the recoverable per-encounter faults — duplicate encounter, multiple/zero claims, orphan rows referencing an unknown encounter, admission-after-discharge — are routed to a quarantine list while clean encounters keep flowing. A max_quarantined circuit breaker aborts a batch that is too broken to trust.

Notional dataexamples/bulk/mercy_regional/ is a synthetic, EHR-warehouse-shaped export (6 encounters, MRN/POA/revenue codes/working-final DRG/ICD-10) used to exercise this whole path end to end. See docs/NOTIONAL_DATA.md.

5Evidence extraction

agent/src/agents/encounter-extractor.ts → investigation.py

The extractor agent receives encounter timing, the ontology contract, and the source documents — never the claim. It returns evidence excerpts, a patient-specific ontology fragment, and clinical assertions (concept, present/absent/uncertain/historical status, explicit/inferred/conflicted/absent documentation status, confidence, supporting & contradicting evidence).

The orchestrator then runs a deterministic gauntlet before anything is trusted:

  1. validate the source bundle;
  2. verify every excerpt is an exact literal substring of the identified document, and its id/author-role/timestamp are unchanged;
  3. validate entity types, relation domain/range, evidence requirements, lineage, and contradictions against the ontology;
  4. attach structural patient/encounter/claim nodes and model/schema provenance outside the model;
  5. merge immutable encounter and claim fields;
  6. run an independent Python validator.

Anything malformed — bad confidence, unknown fields, duplicate ids, naive timestamps, dangling citations, overlapping supporting/contradicting evidence, schema mismatch — fails closed. Grounding is validated in both TypeScript (schema.ts) and Python, with a shared, byte-identical ontology digest so the two runtimes cannot silently diverge. The agent is provider-agnostic via a MODEL_ID string and imports no provider SDK — swapping models changes nothing downstream.

6Ontology & the encounter case

ontology.py, models.py · wound-care-encounter-ontology 1.1.0-draft

The ontology is the semantic backbone everything is built on: ~50 versioned classes (Wound → PressureInjury/DiabeticFootUlcer/…, PressureInjuryStage, DiagnosisCode, Claim, and first-class PresentOnAdmission and DRGSeverityTier), ~36 relations with domain/range, and value-sets (POA status, severity tier, PI stage, exudate, urgency…). It carries a semantic digest that rejects unversioned drift, computed identically in Python and TypeScript.

The EncounterCase (schema 2.0.0) cleanly separates what happened clinically, what was documented, what was coded and charged, what was submitted and paid, and what evidence supports or contradicts a proposed change. New specialties are added as new versioned ontology + rule packages, not engine branches — the same validators serve every domain.

7Governed rules

rules.py, engine.py · declarative, no code execution

Reconciliation logic is authored as versioned, ontology-scoped, declarative rule packages. A rule targets a subject type, matches a condition tree, checks case-level conditions, and proposes a change — but a claim-affecting change always requires human review.

Condition operators

eq · ne · gte · lte · in · contains · not_contains · exists · between · starts_with · count_gte · count_lte

The last four were added to express severity bands, prefix/code logic, and evidence-count thresholds off the ontology.

Derived read-only fields

Python-computed (never model-supplied) on each assertion: evidence_count, contradicting_evidence_count, has_contradicting_evidence — so a rule can decline to propose a change over self-contradicting evidence.

Proposed-change keys are limited to add/remove diagnoses·procedures·charges. Two governed packages ship: wound_care_v1 (omitted-diagnosis + CDI query) and wound_care_v2 (severity coding using the new operators + a hospital-acquired POA compliance rule). Every executable rule carries a subject scope and retains subject/assertion/evidence lineage in its findings; packages fail closed unless approved.

8DRG grouping

grouper.py + governed data/demo_grouping_v1.json

The DRG grouper is the product's core proposition, and it is fully data-driven and versioned rather than hardcoded. A governed grouping-definition artifact declares a base rate, MCC/CC/none severity tiers, and a diagnosis→severity table. The grouper scans the coded diagnoses, resolves the most severe tier deterministically, and prices with integer-cent math.

Every result carries an ordered, hash-stable derivation trace — severity_resolution → tier_selection → pricing — so a reviewer can see exactly why a DRG and payment were produced. It stays a fake (demo-0.2-not-for-billing); a licensed MS-DRG/APR-DRG grouper drops in through the same Grouper protocol.

9Findings, ROI & automation

investigation.py, review_packet.py, automation.py

A Finding binds the rule, disposition, confidence, proposed change, DRG delta, estimated impact, grouper derivation, and full evidence/assertion/subject lineage. From the set of findings the engine computes a deterministic, hash-covered impact summary (recoverable upside, downside at risk, net, counts by disposition) so a CFO can reproduce every figure from the packet.

Exception automation

A separately-hashed automation plan classifies each finding into a tier and minimizes human work:

TierMeaning
suppressedNo opportunity, or an exact semantic duplicate consolidated without double-counting.
needs_enrichmentEvidence insufficient — returns to acquisition rather than disappearing.
auto_routedSupported, high-confidence, bounded-impact operational work placed in a governed queue.
quick_confirmHigh-confidence material DRG change → a prepared action + short confirmation.
focused_reviewLower confidence or documentation clarification needing targeted expertise.
escalatedContradiction, compliance sensitivity, negative or unknown impact — bypasses ordinary budgets.

Priority is an itemized, integer score (tier_weight + confidence_weight + impact_weight + urgency_weight) with an uncapped dollar weight so a six-figure recovery outranks routine work instead of saturating. A reviewer_effort rollup estimates no-touch rate and reviewer-minutes saved (clearly labelled estimates). An idempotent SQLite routing outbox can enqueue only auto_routed items and can never mutate a claim.

10Review packet & audit

The versioned public contract between engine and reviewer

The review packet (3.2.0) is the stable handoff: tenant/workspace scope, the immutable claim snapshot, the evidence graph, findings (with derivation trace + impact), the impact_summary, explicit controls (claim_mutation_allowed = false, permitted actions), and provenance (engine/rule/grouper versions, case & rule-package hashes, a full-packet hash). The packet is a review artifact, not a claim transaction.

Reviewer decisions pass through a role-aware workflow service and a tenant-scoped, hash-linked decision repository; each decision binds the exact packet, plan, actor, structured reason code, and idempotency key. Audit records are canonical-JSON, sha256, and hash-chained — a read-only verify_audit_chain makes the whole ledger tamper-evident and replayable.

11Accuracy & evaluation

evaluation.py, eval_cli.py · make eval

Deterministic discovery is only credible if it is measured. The evaluation harness runs the engine over a labelled gold manifest, maps each finding to an opportunity key, and reports precision / recall / F1 in a canonical, hash-signed report — the honest source for any quoted accuracy figure. A --enforce flag turns threshold breaches into a non-zero exit, so an accuracy regression fails the build. It never touches a claim, DRG, or payment; it only compares engine output to human labels.

12The agentic runtime — two-plane model

src/revenue_integrity/runtime/

To make the agents far more capable without giving up determinism, the platform adopts a two-plane model. The agent authors freely upstream; only verified, versioned, hash-pinned artifacts cross into the deterministic execution plane.

Authoring / design plane

The agent examines bounded data, writes code, runs it in an OS sandbox, evaluates the output against golden samples, and self-corrects — iterating until a deterministic scorer passes.

Execution plane

A verify-then-promote gate admits only a frozen artifact whose hash equals what was scored, with an executable status and a version. Execution loads artifacts from a content-addressed, hash-chained registry.

Net effect: the agent contributes everywhere and writes/runs code freely in the sandbox, while what touches real claims/PHI stays reproducible, versioned, human-approvable, and auditable. Model output is never authoritative until it clears the gate.

13RAG & self-learning

runtime/knowledge.py · retrieval.py · promotion.py

The system improves over time by accumulating verified experience, not opaque model drift:

  • Knowledge store — an append-only, content-addressed, hash-chained ledger of exemplars: approved adapter mappings, transforms, readers, ontology deltas, and labelled reviewer outcomes. Idempotent, tamper-evident, serializable.
  • Deterministic RAG retriever — given a new task's feature tokens (a provider's columns, or a finding's rule + codes), it returns the top-k most relevant prior exemplars by transparent Jaccard overlap with a content-hash tiebreak. Agents get a small, relevant context instead of everything → fewer tokens, higher accuracy, with a reproducible retrieval_digest. It is wired live into the adapter-designer, and pairs with deterministic ontology-subgraph retrieval on the extraction side.
  • Reviewer write-back — a deterministic projection of an audited packet + decision log into retrievable review_outcome exemplars, so future similar findings surface how comparable ones were resolved. Built from the audited artifacts (not a side-effect of submit) and idempotent on replay.

14Sandboxed authoring

runtime/sandbox.py · self_eval.py · authored_reader.py · authored_transform.py · ontology_extension.py

  • OS sandbox — agent-authored code runs in a fresh isolated subprocess (python -I -S, empty env, throwaway cwd, RLIMIT_CPU/FSIZE/NOFILE + RLIMIT_AS on Linux, wall-clock kill). Runaway loops, crashes, and per-row errors are contained and reported — never fatal to the host, and host env vars are invisible to the code.
  • Self-eval loop — score_artifact runs authored code through the same sandbox on golden samples and returns a score (parse rate, conformance, exact match); evaluate_and_admit closes generate → run → evaluate → promote in one call.
  • Authored readers — the agent writes a read(raw) parser for a novel format (FHIR NDJSON, HL7 v2, X12, fixed-width, pipe-delimited…); it is sandbox-scored, promoted hash-pinned, and executed only in the sandbox with a code-hash freeze. Rows flow into the profiler/adapter pipeline.
  • Authored transforms — the same pattern for field logic the fixed DSL can't express: regex extraction, currency→cents, arithmetic, conditional.
  • Ontology-extension gate — the agent proposes an additive-only delta (data, not code); a deterministic preflight fails closed unless it is additive-only, internally valid, version-bumped, and digest-recomputable.
Honest scopeThe sandbox bounds CPU, memory (Linux), files, descriptors, time, and isolates the environment/interpreter. Hard network/syscall isolation is out of scope for this interface and must be added via a container / seccomp jail in production.

15Data model & fidelity

models.py, financial.py

The core case models separate clinical reality, documentation, coding, billing, and evidence. The Claim carries the coded diagnoses/procedures/charges, DRG, and allowed amount — plus optional, additive fidelity fields introduced without breaking existing data:

  • diagnosis_details — per-diagnosis sequence (1 = principal) and present-on-admission indicator, with a principal_diagnosis() accessor;
  • charge_lines — full charge-line detail (revenue code, units, cents, status), activating the financial.ClaimLine model, with a charges_from_lines() reducer.

These are backward-compatible (legacy cases are unaffected). A formal case schema_version migration and a sequence-/POA-aware grouper are the natural follow-ups.

16Formats & known limits

What flows through today, and where the edges are.

Native readers: CSV, JSON, JSONL, XLSX. Via authored readers (sandbox-verified): arbitrary text formats such as FHIR NDJSON, HL7 v2, X12 837/835, fixed-width, pipe-delimited. Not yet handled: OCR/layout for PDF/TIFF/scans, and native binary/columnar formats (Parquet/Avro).

Two honest, structural trade-offs to name for any evaluator:

  • Safety vs. coverage. Declarative + fail-closed + verified-code is what makes the platform auditable and non-hallucinating — but it caps how much messy data flows through without human/agent authoring. The quarantine plane softens whole-batch failure into per-encounter degradation.
  • Fidelity vs. the synthetic core. The engine, rules, ontology, and agentic runtime are real; the grouper, terminology, and pricing are demo-grade until a licensed grouper and a terminology service are attached.
