A reference system that reconstructs an inpatient encounter, compares what was documented against what was coded and billed, and routes only the consequential exceptions to a human — while a language model is kept strictly on the outside of every authoritative decision.

Core invariant

No language-model output can execute code, create or change a claim, assign a DRG, calculate reimbursement, or bypass required review. The model emits evidence and hypotheses; deterministic software decides everything else.

01 / 09 Why this is built the way it is

Clinical text wants a model.
Money wants a machine.

Reading a messy clinical note — inferring that a wound is a stage-4 pressure injury — is exactly what a language model is good at. Deciding a DRG, simulating a payment, or altering a claim is exactly what a model must never do unsupervised.

So the two jobs are physically separated. The Mastra agent only produces schema-constrained evidence excerpts and hypotheses. The Python engine stays reproducible and model-independent: same input, same output, forever auditable.

The model is provider-agnostic — selected by a single provider/model string (MODEL_ID). The application imports no vendor SDK. Swapping the model changes the extraction quality and nothing else: not the ontology, not the rules, not the numbers.

Every production assertion also records the model, prompt, extraction policy, and terminology versions — so any accepted result can be reproduced under the exact boundary that produced it.

03 / 09 The trust boundary, concretely

What the model is handed — and what it is never allowed near

During evidence extraction, the agent receives encounter timing, an ontology contract, and the source documents. It returns excerpts, a patient-specific ontology fragment, and clinical assertions. It is structurally blind to the claim.

In What the model receives & returns

  • Encounter timing window
  • Ontology contract (classes, relations, digest)
  • Source documents (notes, narrative text)
  • A bounded profile — for adapter design only
  • → returns exact evidence excerpts
  • → returns a patient ontology fragment
  • → returns clinical assertions + confidence

Never What the model cannot see or emit

  • The claim, charges, or existing DRG
  • Any payment or reimbursement field
  • Executable rules or generated code
  • Source locators / provenance (attached by software)
  • Authoritative financial fields
  • Structural patient / encounter / claim nodes
  • Any path that mutates or submits a claim

05 / 09 The fail-closed gate

Eight checks stand between the model and the engine

When the agent returns, the orchestrator does not trust a single field. It verifies grounding, merges immutable data outside the model, and runs an independent Python validator before anything is evaluated.

Validate the source bundle.

Verify every excerpt is an exact literal substring of its source.

Verify document ID, author role, and timestamp are unchanged.

Validate entity types, relation domain/range, evidence requirements, lineage, and contradictions.

Attach structural patient, encounter, and claim nodes — outside the model.

Attach model and schema provenance — outside the model.

Merge immutable encounter and claim fields.

Pass the completed case through an independent Python validator.

FAIL-CLOSED → invalid confidence · unknown fields · duplicate IDs · naive timestamps · dangling citations · overlapping supporting/contradicting evidence · schema-version mismatch — all reject the payload. Any proposed claim change requires human review.

07 / 09 Exception automation

People decide only what is worth deciding

A separately hashed automation plan binds the exact packet and assigns each finding a tier. Duplicates and disproven hypotheses never reach a human; genuine, ambiguous, or compliance-sensitive exceptions do.

suppressed
Exact duplicate or no supported opportunity — cleared automatically, no human touch.
needs_enrichment
Evidence is insufficient — the system requests more before anyone reviews.
auto_routed
Bounded operational work is queued via an idempotent outbox that cannot mutate claims.
quick_confirm
A person makes a fast, low-ambiguity confirmation.
focused_review
A true exception — material, ambiguous, or institutionally controlled — gets a full packet.
escalated
Conflict or compliance sensitivity raises priority rather than authorizing a code.

Routing targets are ordinary work queues — coding · cdi · charge · compliance. Agent disagreement is treated as supporting evidence that increases escalation priority, never as consensus truth.

09 / 09 Running it yourself

Three toolchains, one gate

The deterministic demo

No model credential needed. Builds a finding, evidence, a proposed change, a demo regrouping and payment delta — without touching a claim.

# full local gate: tests + typecheck + demo
make verify
make demo

The model layer

Provider-agnostic. Change the model with an env var; the ontology and engine are untouched.

# pick any Mastra-supported provider/model
MODEL_ID=anthropic/<model> \
  npm run extract -- bundle.json out.json ontology.json

Bulk onboarding

Profile a provider folder, then run an approved deterministic adapter over the full dataset. Only the bounded profile ever reaches the adapter-designer agent.

revenue-integrity-ingest profile examples/bulk/clinic_alpha --output profile.json
revenue-integrity-ingest run examples/bulk/clinic_alpha adapter.json --output-directory bundles/

What this is not

Not for production billing or clinical use. Every rule, code, price, patient, facility, and the demo grouper is synthetic and unvalidated. Real deployment still requires licensed terminology and grouping components, FHIR/HL7 and claim adapters, institution-approved rules, representative validation data, model evaluations, a reviewer application, and the documented security controls.
